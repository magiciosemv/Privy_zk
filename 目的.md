## 1. 项目名称与定位

**名称：** Privy SVM  
**全称：** Programmable Privacy Virtual Machine on Solana  
**一句话：** 让 Solana 程序拥有一行注解即可获得零知识隐私的能力。

**定位：** Solana 生态的隐私基础设施层。不是单个应用，不是单个协议，而是一个**平台**——任何 Solana 开发者都可以在自己的程序里接入隐私能力，就像今天接入 SPL Token 一样简单。

**目标用户：**
- 写 Solana 程序的 Rust 开发者（不再需要学 Noir/Circom）
- DeFi 协议方（给用户提供隐私交易选项）
- GameFi 开发者（保护游戏隐藏状态）
- DAO 治理（隐私投票、隐私薪酬）

## 2. 核心功能

### 2.1 `#[private]` 注解 —— 一行代码获得隐私

```rust
use privy_svm::prelude::*;

#[program]
mod my_private_program {
    #[private]               // ← 只有这一行
    let balance: u64;        // balance 现在在链上不可见

    #[private]
    let credit_score: u16;   // 信用分也是私有的

    pub valid_total: u64;    // 没标注的就是普通公开状态

    pub fn transfer(amount: u64) {
        // 你可以用零知识证明证明 balance >= amount
        // 但链上看不到 balance 或 amount 的具体值
    }
}
```

**开发者不需要：**
- 学 Noir（Aztec 的隐私语言）
- 学 Circom（ZK 电路语言）
- 理解 Groth16/Plonk/STARK 的数学
- 手动写 R1CS 约束矩阵

**开发者只需要：**
- 会写 Rust（本来就要写 Solana 程序）
- 在变量上标注 `#[private]`
- 运行 `privy build` → 自动生成 ZK 电路 + Verifier + SDK

### 2.2 字段级隐私粒度

这不是合约级隐私（整个程序所有状态都是私有的），也不是交易级隐私（只隐藏交易金额）。Privy SVM 支持的是**字段级**：

```
struct UserAccount {
    pub username: String,         // 公开 ✅
    #[private]
    pub encrypted_balance: u64,   // 私有 🔒
    pub join_date: i64,           // 公开 ✅
    #[private]
    pub kyc_status: bool,         // 私有 🔒
    #[private(selective)]
    pub country: String,          // 用户可选择是否披露 🔒→✅
}
```

同一个账户里，公开字段照常可用，私有字段零知识保护。这是目前所有隐私方案中粒度最细的。

### 2.3 选择性披露

```rust
#[private(selective(credit_gt_700, is_verified))]
let credit_info: CreditData;

// 用户可以生成证明：
// "信用分 > 700" ✓   → 只公开这个布尔值
// "已通过 KYC"  ✓    → 只公开这个布尔值
// credit_info 的原始数据 → 永远不暴露
```

贷款协议拿到两个布尔值就能放款，不需要知道用户的真实信用分数、地址、交易历史。

### 2.4 可撤销隐私

```rust
#[private(revocable)]
let access_grant: TokenID;

// 用户随时可以：
// privy revoke --nullifier 0xabc...
// 所有依赖此证明的协议都会收到链上事件通知
```

解决"隐私 = 永久"和"合规 = 可撤销"的矛盾。符合 GDPR "被遗忘权"标准。

### 2.5 跨程序隐私组合调用（核心超级能力）

```
公开 DEX 程序 ─── CPI 调用 ───▶ 隐私信用程序 ─── 返回 ZK 证明 ───▶ 公开借贷协议
   "1 SOL 换 USDC"               "检查信用"          "信用 > 700"      "批准 1000 USDC 借款"
                                 (不暴露分)        (不暴露身份)
```

一切在同一条链上、同一个事务中完成，原子性由 Solana CPI 原生保证。不需要跨链桥，不需要异步通信，不需要信任第三方中继器。这是任何其他隐私方案（Aztec、Tornado Cash、Elusiv）都做不到的。

### 2.6 ZK 电路自动编译器（PrivyCC）

```
Rust 源码 + #[private] 注解
        │
        ▼
   注解解析器（proc-macro）
        │
        ▼
   中间表示 IR（提取所有私有变量和约束）
        │
        ▼
   自适应路由（自动选择 Groth16 / Plonk / STARK）
        │
        ▼
   自动编译为：
   ├── Solana Verifier Program（链上验证用）
   ├── Proving Key + Verification Key
   └── Client SDK（浏览器 WASM / Rust 原生）
```

## 3. 它是什么，不是什么

| 是 | 不是 |
|---|---|
| **平台/基础设施** — 任何 Solana 程序都可以接入 | ❌ 单个应用或协议 |
| **字段级隐私** — 精确到每个变量 | ❌ 合约级或交易级粗粒度 |
| **Rust 原生** — 用 Rust 注解，零新语言 | ❌ 要求学 Noir/Circom |
| **选择性披露** — 用户可以控制披露什么 | ❌ 全匿名混币器 |
| **可撤销** — 符合 GDPR，支持合规审计 | ❌ 不可逆的永久隐私 |
| **Solana 原生** — 利用并行执行和低手续费 | ❌ 跨链桥或 L2 |

## 4. 解决的痛点 —— 为什么 Solana 急需这个

### 痛点 1：Solana 链上数据 100% 公开

**现状：** Solana 的每一个程序、每一个账户、每一笔交易的每一个字节，都是公开可见的。这是设计上的取舍——高性能的代价是零隐私。

**具体影响：**

- **鲸鱼地址被全程监控。** 大户钱包的每一笔交易都被 front-run bot 实时监控。一笔大额买入还没上链，MEV 机器人的买单已经排在你前面。据估算，Solana DeFi 用户每年因 MEV 损失超过 5000 万美元。
- **DeFi 持仓和策略完全暴露。** 你的止损位公开在链上，你做市资金规模公开在链上，你的收益策略公开在链上。竞争对手可以精确分析你的每一个动作。在传统金融里这叫"内幕信息"，在 Solana 上叫"区块链透明性"。
- **做市商和机构无法使用 Solana。** 没有任何机构会把国库管理、做市策略、持仓分布放在一条完全透明的链上。这直接限制了 Solana 上 DeFi 的 TVL 天花板。

### 痛点 2：因缺乏隐私而完全不可行的应用类别

这不是"体验不好"的问题，而是**整个品类被锁死了**：

**暗池交易：** 大宗交易需要隐藏挂单直到成交，否则滑点成本 2-5%。传统金融有暗池，中心化交易所有冰山订单。Solana 上没有——因为链上挂单数据全部公开。

**链上博彩/策略游戏：** 扑克手牌必须隐藏。你的底牌如果公开在链上，这不是游戏，是数学题。目前所有 Solana 上的游戏要么放弃博彩市场（数十亿美元规模），要么妥协做中心化服务器存储。

**隐私 DAO 投票：** 你的投票内容如果公开，买票、胁迫、跟风投票就无法阻止。DAO 投票的有效性直接取决于投票隐私。目前几乎所有 DAO 都在用中心化投票工具（Snapshot）解决这个问题——因为链上投票没有隐私。

**隐私薪酬/工资流：** DAO 成员的薪水如果公开在链上，团队管理就是噩梦。不同地区、不同贡献者的薪酬差异会引发无穷争议。目前 100% 的 DAO 团队用中心化工具（Utopia/Parcel 等）处理薪酬。

**链上信用评分：** 传统金融可以查询你的信用分而不暴露你的银行流水。区块链只能"透明"——要证明你的链上信用，就得暴露所有交易历史。这完全不可接受。

**企业国库管理：** 没有企业会在 Solana 上公开管理几千万美元的国库。因为你的持仓、现金流、对手方全部暴露，竞争对手可以做精准打击。

### 痛点 3：隐私和合规的冲突

**Tornado Cash 被制裁的原因：** 它是一个完全的匿名混币器——任何人都可以匿名存取，包括黑客和洗钱者。监管无法区分合法用户和非法用户。

**市场需要的是"选择性披露"而不是"完全匿名"：**
- 我需要证明"我不是洗钱者"，但不暴露我的交易记录
- 我需要证明"我通过 KYC"，但不暴露我的护照号
- 我需要证明"我信用够好"，但不暴露我的银行流水
- 我可以随时撤销我之前发布的证明（被遗忘权）

**目前 Solana 上没有这种中间地带。** 要么完全透明，要么去用 Tornado Cash 那类被制裁的工具。没有任何方案让用户"有选择地"披露信息。

### 痛点 4：开发者门槛

如果你是一个 Solana 开发者，想在程序里加隐私功能，今天你的选择是：

| 方案 | 要做的事 | 成本 |
|---|---|---|
| 自己写 ZK 电路 | 学 Circom/Noir + Groth16/Plonk 数学 + 写 R1CS 约束 + 调试 + 安全审计 + 部署 | 6-12 个月 |
| 用 Light Protocol | 学 ZK 压缩原语，自己搭电路 | 2-4 个月 |
| 去 Ethereum + Aztec | 学 Noir 语言 + Solidity 迁移 + L2 bridge | 3-6 个月 + 离开 Solana |
| **Privy SVM** | 在 Rust 变量上标注 `#[private]` | **5 分钟** |

这种门槛差异，决定了今天 Solana 生态几乎没有通用的隐私程序。

## 5. 创新点详解

### 创新 1：Rust 注解驱动的 ZK 编译器

**以前是怎么做的：**

开发者需要先学一门专门为 ZK 设计的语言（Noir、Circom、ZoKrates），然后手动把程序逻辑翻译成 ZK 电路。这要求理解：R1CS 约束系统、多项式承诺方案、椭圆曲线配对、可信设置等底层密码学。

**Privy SVM 的做法：**

开发者写 Rust 代码，在变量上加 `#[private]` 注解。编译器（proc-macro）自动：
1. 扫描所有 `#[private]` 标注的变量
2. 分析所有涉及这些变量的条件判断和状态转换
3. 提取为中间表示（IR）
4. 根据约束规模自动选择 Groth16 / Plonk / STARK 后端
5. 生成 ZK 电路 + 验证密钥 + 链上验证器 + 客户端 SDK

**类比：** 就像 Rust 的 `#[derive(Serialize)]` 自动生成序列化代码。你写 `struct User { name: String, age: u8 }` 加一行注解，编译器自动生成序列化实现。Privy SVM 做的是同样的事情——你写 Rust 程序逻辑，加 `#[private]` 注解，编译器自动生成 ZK 电路。

**为什么这是革命性的：** 它把 ZK 开发从"密码学专家才能做的事情"变成了"任何 Solana 开发者都会做的事情"。这是 ZK 技术的"LLVM 时刻"——不需要每个人理解底层编译原理。

### 创新 2：字段级隐私粒度

**现有方案的局限：**

- Tornado Cash / Elusiv：交易级——只能隐藏交易金额和参与方
- Aztec：合约级——整个合约的所有状态都是私有的
- Light Protocol：API 级——提供 ZK 原语但不提供隐私语义

**字段级的含义：**

一个程序里，你可以精确地控制哪些字段公开、哪些字段私有。这不是"全公开"或"全隐私"的二选一，而是每个变量独立控制。

```
struct GameState {
    pub board: [u8; 64],        // 棋盘状态公开（大家都在看棋盘）
    #[private]
    pub my_hand: [Card; 5],     // 我的手牌私有（对手不可以看到）
    #[private]
    pub my_chips: u64,          // 我的筹码私有（不暴露策略）
    pub pot: u64,               // 底池公开（大家都在争）
    pub current_bet: u64,       // 当前下注公开
    #[private(selective)]
    pub folded: bool,           // 我可以选择不暴露是否弃牌直到摊牌
}
```

**为什么这很重要：**
真实应用需要混合隐私。完全公开破坏用户体验，完全隐私破坏可审计性。字段级让你能在同一个交易里兼顾两者。

### 创新 3：原生混合调用（公开↔隐私跨程序组合）

**现有方案的局限：**

在 Ethereum 上，如果你想让公开合约和隐私合约交互：
1. 公开合约在 Ethereum L1
2. 隐私合约在 Aztec L2
3. 需要跨 L2 桥接
4. 异步通信，有延迟，有安全风险
5. 不是原子性的

**Privy SVM 的做法：**

```
公开程序 → Solana CPI → 隐私验证器 → ZK 验证通过 → 公开程序继续执行
```

因为一切都发生在 Solana 的同一条链上，Solana 的 CPI（跨程序调用）机制原生保证原子性。整个调用链要么全部成功，要么全部回滚。没有桥接、没有跨链、没有异步等待。

**为什么这是决定性的差异化：**
它让一个 DeFi 协议（完全是公开逻辑）可以在内部集成隐私能力，而不需要改变自己的架构。Aave 在 Solana 上的版本可以无缝调用 Privy SVM 来验证用户的隐私信用评分，然后决定是否放贷——全程同步，毫秒级。

### 创新 4：可撤销隐私 + GDPR 原生支持

**隐私的"黑暗面"：**

传统的隐私方案一旦发布就无法撤销。你发布了一个证明"我不是美国人"，5 年后你搬到了美国——证明仍在链上存在，无法删除。

这产生了两个问题：
1. GDPR 的"被遗忘权"要求数据可以被删除
2. 合规审计需要能够撤销特定权限

**Privy SVM 的解决方案：**

每个隐私证明都有一个 nullifier。用户可以通过签名操作撤销这个 nullifier，所有依赖此证明的协议都会收到链上事件通知。时间窗口过期机制让证明可以自动失效。

```
struct KYCProof {
    verified: true,
    #[private(revocable)]
    nationality: String,       // 用户可随时撤销
    #[private(expires = "30d")]
    verification_date: i64,    // 30 天后自动过期
}
```

### 创新 5：自适应性 ZK 后端路由

**三种 ZK 方案有各自的优劣：**

| 方案 | 证明大小 | 验证速度 | 可信设置 | 适用场景 |
|---|---|---|---|---|
| Groth16 | 128B | 极快 | 需要 | 小计算、高频交易 |
| Plonk | 400B | 较快 | 不需要 | 中等计算、频繁升级 |
| STARK | 10KB+ | 较慢 | 不需要 | 大计算、高吞吐 |

**Privy SVM 的做法：**

编译器分析程序的约束数量、选择性披露需求、是否频繁升级等维度，自动选择最优 ZK 后端。开发者不需要做这个选择——编译器根据实际程序特征自动做最佳决策。

约束 < 500 → Groth16  
约束 500-5000 + 频繁变更 → Plonk  
约束 > 5000 → STARK

这就像 Rust 编译器的优化等级选择一样——你不需要手动指定 `-O2` 还是 `-O3`，编译器自己知道。

## 6. 与竞品的本质区别

|  | Aztec | Tornado Cash | Elusiv | Light Protocol | **Privy SVM** |
|---|---|---|---|---|---|
| **链** | ETH L2 | ETH | Solana | Solana | **Solana** |
| **隐私粒度** | 合约级 | 交易级 | 交易级 | API级 | **字段级** |
| **语言** | Noir | 无 | 无 | Rust原语 | **Rust注解** |
| **学习门槛** | 高（新语言） | 低（只用） | 低（只用） | 高（自搭电路） | **极低（一行注解）** |
| **公开↔隐私混合调用** | 需要桥接 | 不支持 | 不支持 | 不支持 | **原生CPI原子调用** |
| **选择性披露** | 有限 | 不支持 | 不支持 | 不支持 | **完全支持** |
| **可撤销** | 不支持 | 不支持 | 不支持 | 不支持 | **支持+过期** |
| **合规** | 匿名混币风险 | 被制裁 | 被审查 | 中立 | **合规隐私混合** |
